Simplicity

"Simplicity is prerequisite for reliability."
— Edsger W. Dijkstra
"There are two ways of constructing a software design: One way is to make it so simple that there are obviously no deficiencies and the other way is to make it so complicated that there are no obvious deficiencies."
— C. A. R. Hoare
"Simple things are those which have one role."
— Rich Hickey

Simple != easy.

Working Definition

Simplicity is the disciplined removal of accidental entanglement so that a system can be understood, reasoned about, changed, and trusted.

It is not minimalism, familiarity, or clever brevity. It is hard-won clarity.

Modernization

Changing an existing system so it can keep serving the business under today's constraints.

Port

Same external behavior, new technical substrate.

Rewrite

Preserve the business capability, but change design boundaries and implementation to reduce accidental complexity.

🚀 Port

TypeScript Native

Same language and tooling experience, new native implementation substrate.

devblogs.microsoft.com

Why Modernize?

  • Large TypeScript codebases need faster editor startup and type checking.
  • The goal is better developer experience at scale.
  • Microsoft reported about 10x faster builds, faster editor startup, and lower memory usage.
  • The implementation target is Go.

Process

  • Port the compiler and tools to native Go.
  • Preview command-line type checking first.
  • Move toward project builds and language service parity.

Safety Mechanism

  • Behavior parity is the product promise.
  • TypeScript 6 remains the JavaScript line.
  • TypeScript 7 becomes native once mature enough.

Outcome

What worked What to watch
Clear technical target: faster tooling. Compatibility and API parity must be protected.
Reported order-of-magnitude speedups on large projects. Existing projects may remain on TypeScript 6 until TypeScript 7 fits their constraints.
Go supports the operational goal with native binaries and strong tooling. The port succeeds only if behavior remains familiar.

🚀 Rewrite

Reddit Comment Backend

Incremental migration of a high-throughput write path from Python to Go.

reddit.com/r/RedditEng

Why Modernize?

  • Comments are a high-throughput, user-visible write path.
  • The target was lower latency without user disruption.
  • The new service was written in Go.

Process: Tap Compare for Writes

sequenceDiagram actor Client participant Go as Go Service (new) participant Py as Python Service (legacy) participant PD as Production Stores
(Postgres + Memcached + Redis) participant SD as Sister Stores
(isolated — Go only) Client->>Go: Write request (small % of traffic) Go->>Py: Delegate real write Py->>PD: Write to production Py-->>Go: Response Go->>SD: Write independently (no production impact) Go->>Go: Compare production vs sister stores Note over Go: Log differences — never return new response to user Go-->>Client: Return legacy response

Safety Mechanism

  • Tap compare for write-path behavior validation.
  • Sister datastores for write-path validation.
  • The legacy response remained the user-facing truth during validation.

Outcome

What worked What to watch
Zero user disruption during migration. Shadow validation does not remove the need for strong local tests.
p99 write latency was reduced by about 50%. Raw Go queries exposed DB pressure hidden by the Python ORM.
Sister datastores protected production writes. Race conditions created false tap-compare mismatches.
Tap compare gave production confidence before cutover. Finding edge cases through production logs is expensive.

🚀 Rewrite

Turso Limbo

A SQLite-compatible rewrite in Rust, designed with reliability testing from day one.

turso.tech/blog

Why Modernize?

  • libSQL started as a SQLite fork and reached changeability limits.
  • Some key SQLite test infrastructure, such as TH3, is proprietary.
  • C made invasive evolution riskier for the target changes.

Process

  • Rewrite SQLite-compatible behavior in Rust.
  • Design for async I/O from the beginning.
  • Keep compatibility visible by verifying generated bytecode against SQLite.
  • Regularly fuzz inputs to find edge cases.

Safety Mechanism

  • Deterministic Simulation Testing built in from day one.
  • Failures can be reproduced reliably.
  • Antithesis fault injection found rare io_uring partial-write failures.

Outcome

What worked What to watch
Reliability strategy was part of the architecture from day one. A full rewrite needs exceptional justification.
DST made rare failures reproducible. Compatibility with SQLite behavior remains a large burden.
Antithesis found failure modes internal tests may have missed. The testing investment is not optional; it is the price of the rewrite.

🚀 Rewrite

Slack Desktop

The Ship of Theseus: module-by-module replacement while continuously shipping.

slack.engineering

Why Modernize?

  • The desktop app had accumulated years of technical debt.
  • Performance and maintainability were degrading at scale.
  • A big-bang rewrite would have frozen value delivery for years.

Process

  • Replace one module at a time while shipping to users throughout.
  • The first modern component — the emoji picker — shipped more than two years before the full rollout.
  • No cutover moment: the app was always in production.

Safety Mechanism

  • Continuous shipping meant continuous user feedback.
  • Each module replacement was independently releasable and reversible.
  • No dependency on a "when it's done" moment.

Outcome

What worked What to watch
Users experienced continuous improvement, not a big reveal. Module boundaries must be identified carefully to allow independent replacement.
Risk was distributed across many small releases. Old and new code coexist for a long time — boundary contracts must be maintained.
Value delivered from day one, not after years of tunnel work. Team discipline on the new patterns must be sustained across the long transition.

Success

Releasing incrementally delivered real value, maintained continuous improvement focus, and de-risked the release.

🚀 Rewrite

Dropbox Nucleus

A rare justified big-bang rewrite — when the data model is structurally broken.

dropbox.tech

Why Modernize?

  • The sync engine was built before file sharing existed.
  • Files had no stable identifier preserved across moves — sharing could not be added incrementally.
  • The data model was the problem; no amount of patching could fix a wrong foundation.

Process

  • Four-year rewrite of the desktop sync engine from Python to Rust.
  • A decision checklist applied before starting: incremental improvements exhausted, resources confirmed, slower feature velocity accepted.
  • Clear technical goal: a new data model with stable file identifiers, not just a language swap.

Safety Mechanism: The Checklist

  • Are incremental improvements truly exhausted?
  • Do you have the resources and organisational health?
  • Can the team accept slower feature development for the duration?

Outcome

What worked What to watch
Rust became a force multiplier — described as one of the best decisions made. Four years is a long commitment; the business must be able to sustain it.
The new data model correctly supported sharing from its foundation. A big-bang rewrite is only justified when the structure itself is wrong, not just the implementation.
The checklist gave the team an explicit go/no-go framework. Feature development was slower for the duration — this cost must be accepted up front.

Success

A big-bang rewrite is justified only when incremental improvement is structurally impossible.

🚀 Decomposition

Shopify Monolith

DDD-guided decomposition into a modular monolith — not microservices.

shopify.engineering

Why Modernize?

  • The Rails monolith needed decomposition for developer velocity at scale.
  • Microservices were the obvious choice but increased overall complexity considerably.
  • Clear ownership boundaries were needed without distributing the system.

Process

  • Use DDD subdomains to identify decomposition boundaries.
  • Implement boundaries as Rails Engines within the existing monolith.
  • Vision-guided approach: DDD first, then engineering implementation.
  • 37 components with public entrypoints established.

Safety Mechanism

  • Modular monolith avoids the complexity of distributed systems.
  • Explicit public entrypoints make cross-boundary coupling visible.
  • Decomposition is incremental; the monolith stays deployable throughout.

Outcome

What worked What to watch
DDD subdomains gave teams clear ownership without distributed complexity. Vision-guided decomposition takes longer than spin-off — teams must accept that timeline.
Rails Engines kept the monolith deployable while partitioning accelerated. Complete boundary enforcement requires sustained discipline across a large developer base.
Explicit public entrypoints made accidental coupling visible. Boundaries evolve as the domain is better understood — 37 components is not the end state.

Success

DDD subdomains gave Shopify clear ownership without the complexity of microservices.

🚀 Migration

Stripe Online Migrations

Migrating hundreds of millions of records with no downtime window.

stripe.com/blog

Why Modernize?

  • The Subscriptions data model needed structural changes at very large scale.
  • A maintenance window was not an option — Stripe processes payments continuously.
  • Data consistency must be guaranteed at every step of the migration.

Process: Four-Phase Dual Write

1. Dual write
Write to both old and new tables simultaneously.
2. Change read paths
Switch all reads to the new table; old table still written.
3. Change write paths
Write exclusively to new table; stop writing old.
4. Remove old data
Delete old table once new is verified as sole source of truth.

Safety Mechanism

  • Each phase is independently verifiable before proceeding to the next.
  • Each phase is reversible — no single commit point.
  • Consistency is maintained throughout: no moment where data exists only in one place.

Outcome

What worked What to watch
Hundreds of millions of objects migrated with zero user impact. Each phase boundary requires careful verification before advancing.
No maintenance window required — migration ran in production continuously. Dual-write adds write amplification; monitor DB load during phase 1.
Each phase could be paused or rolled back independently. Schema changes may require additional coordination beyond the four phases.

Success

No cutover moment. Every phase is independently verifiable and reversible.

🚀 Port

Khan Academy — Goliath

One million lines of Python 2 migrated to Go, without scope creep.

blog.khanacademy.org

Why Modernize?

  • Python 2 reached end of life — ~1 million lines needed to move.
  • The rewrite was also an opportunity to move toward a services architecture.
  • Scope creep was the primary risk: rewrites that add features while porting rarely finish.

Process

  • Direct port from Python to Go — no new features during migration.
  • A single decision-maker controlled whether any net-new Python features were allowed.
  • Project ran from early 2020; reached 95% traffic on the new backend by August 2021.

Safety Mechanism

  • Strict "direct port only" policy enforced by a single gatekeeper.
  • Incremental traffic ramp: new backend handled progressively more traffic before full cutover.
  • Scope discipline: no business logic changes mixed with the port.

Outcome

What worked What to watch
The port finished — scope control was the deciding factor. A million-line port requires sustained organisational commitment across 18+ months.
Go's static tooling and explicit error handling made the new codebase easier to reason about. Mixing port work with new feature development is a reliable way to never finish.
95% of traffic shifted before any public announcement. Incremental cutover requires parallel infrastructure costs during transition.

Success

Scope discipline — not language choice — is what makes a large port finish.

🚀 Rewrite

Airbnb SOA Migration

A Rails monolith decomposed into Java services, one endpoint at a time.

infoq.com

Why Modernize?

  • The Rails monolith could not scale to Airbnb's deployment pace.
  • Hour-long deploy cycles blocked the team from shipping continuously.
  • Different parts of the system had different scaling and ownership needs.

Process

  • Extract one endpoint at a time from the monolith into a Java service.
  • Use dual-read/dual-write with configurable feature gates per endpoint.
  • Start at 1% traffic comparison, ramp to 100% before cutover.

Safety Mechanism

  • Feature gates allow per-endpoint rollback without a full deployment.
  • Traffic comparison at every ramp step catches behavioral divergence early.
  • Endpoint-by-endpoint extraction keeps the monolith as a live fallback.

Outcome

What worked What to watch
Deploy frequency grew to around 10,000 deploys per week. Distributed systems add latency and operational complexity — monitor carefully.
Search page became approximately 3× faster; home description page over 10× faster. Dual-write at scale requires careful consistency discipline across both paths.
Incremental extraction kept the monolith serving users throughout. Service proliferation requires investment in platform and observability tooling.

Success

One endpoint at a time, with a live fallback at every step.

💥 Migration

TSB Bank

A big-bang data migration that locked out 1.9 million customers.

icedq.com

Summary

  • April 2018: 5 million customers migrated to a new platform in a single weekend cutover.
  • 1.9 million customers could not view their accounts after go-live.
  • The team simultaneously upgraded the platform version and migrated the data.

Process

  • Full cutover in a single maintenance window
  • platform upgrade and data migration combined into one event
  • testing ran against an incomplete system

Safety Mechanism

  • No staged rollout or incremental ramp
  • no fallback path once cutover began
  • testing was incomplete because the platform was still being developed during QA

Outcome

What was attempted What failed
Migrate all customers in one weekend 1.9 million customers locked out; some for weeks
Upgrade platform and migrate data simultaneously Missing data scenarios in production that tests never exercised
Complete QA before go-live Testing against an incomplete platform left unknown gaps

Failure

Never test against an incomplete system, and never do everything at once.

Success factors

What made the difference?

  • 77% — No tunnel — always in production, continuous feedback from users
  • 54% — Each step was independently reversible before the next began
  • 38% — Shadow validation kept the legacy response authoritative during migration
  • 38% — A go/no-go checklist before committing to a big-bang rewrite
  • 31% — Reliability testing designed in from day one, not bolted on later
  • 31% — Scope controlled by a gatekeeper; port and new features never mixed
  • 8% — DDD subdomains gave clear ownership without distributing the system

Failure factors

What made the difference?

  • 46% — Long coexistence erodes boundary contracts without sustained discipline
  • 38% — Big-bang removed the user feedback loop for too long
  • 38% — Parity underestimated — scope crept until the market window closed
  • 38% — Compatibility burden is permanent, not a one-time cost
  • 23% — Combining too many changes in one event left no fallback path
  • 15% — New codebase, same habits — the rewrite did not fix what produced the mess
  • 15% — Service proliferation requires platform investment that is often skipped

What made the difference?

The top success patterns are one idea at three levels: never be in a position where you can't see what's happening or can't go back.

The top failure patterns share one enemy: time. Long coexistence, long tunnels, and underestimated parity all fail the same way — the project outlasts the team's ability to sustain it.

The successful teams weren't smarter. They structured their work so that reality corrected them continuously, rather than all at once at the end.

DDD helps us keep rewrites simple

  • Align domain experts, devs and application.
  • Makes sense to the business business sf === good investment
  • Teach the business about itself
  • Define the business capability: use ubiquitous language and domain conversations to identify what must be preserved or improved.
  • Find domain seams: use bounded contexts, EventStorming, and consistency boundaries to choose a migration slice.

Avoid heavy DDD content such as CQRS and event sourcing in the main path.

DDD - Discover — Event Storming

A collaborative workshop: post domain events on a timeline, then work backwards to commands and aggregates.

graph LR C1[BorrowBook] --> E1[BookBorrowed] C2[ExtendLoan] --> E2[LoanExtended] C3[ReturnBook] --> E3[BookReturned] C3 --> E4[LoanOverdue] C4[ReserveBook] --> E5[BookReserved] style E1 fill:#f90,color:#000 style E2 fill:#f90,color:#000 style E3 fill:#f90,color:#000 style E4 fill:#f90,color:#000 style E5 fill:#f90,color:#000

Orange = domain event. Blue = command. Start with events — they surface what the business actually cares about.

DDD - Domains, Sub-domains & Bounded Contexts

  • Domain — the whole business problem space.
  • Subdomain — a coherent area of expertise, discovered during event storming.
  • Bounded Context — explicit boundary where a model is self-consistent. Member in Lending ≠ Member in Notifications.
Bounded Context Owns Language sample
Lending Loan, Copy, Member borrow, extend, overdue
Catalog Title, Edition, Copy acquire, catalog, withdraw
Notifications Recipient, Message notify, reminder, channel

DDD - Context Maps

A context map makes inter-context coupling explicit and names the integration pattern.

graph LR Lending -->|"Customer / Supplier"| Catalog Lending -->|"Conformist"| Notifications Lending -->|"ACL"| Legacy["Legacy ILS"] style Lending fill:#1a6,color:#fff style Legacy fill:#666,color:#fff

ACL = Anti-Corruption Layer: translates a foreign model at the boundary before it can corrupt your domain.

DDD - Message Types

Events from the workshop fall into three distinct roles that shape how the model is structured.

Type What it is BorrowBook example
Command An intent. May be rejected. BorrowBook{MemberID, CopyID}
Event A fact. Immutable, past tense. BookBorrowed{LoanID, At time.Time}
Query A read request. No side effects. ActiveLoansByMember{MemberID}

Commands and queries enter the domain from outside. Events flow out — or drive consistency between aggregates.

DDD - Strategize

Classify subdomains to decide where to invest design effort.

Type What it means Investment BorrowBook
Core Your competitive advantage Own it. Design with care. Lending
Supporting Needed but not differentiating Build simply or buy. Catalog
Generic Commodity Buy or use open source. Notifications

DDD investment follows the core domain. Don't apply heavy tactical patterns to generic subdomains.

DDD - Code Architecture

The bounded context becomes the module. Hexagonal Architecture wires it to the outside world.

graph TD HTTP["HTTP: BorrowBook command"] --> App["Application: BorrowBookHandler"] App --> Dom["Domain: Loan, Member, Copy, DueDate"] App --> Port["LoanRepository port"] Port --> DB["DB Adapter"] Dom --> Event["BookBorrowed event"] style Dom fill:#1a6,color:#fff style App fill:#17a,color:#fff

Architecture is not DDD. DDD gives you the model; hexagonal gives you the wiring. Dependencies always point inward.

The simplest target architecture

  • Start with a domain-oriented modular monolith.
  • Deploy together until splitting is proven necessary.
  • Evolve toward 4-12 coarse services with a simple template when separation pays for itself.
  • Do not start with microservices by default.

Go helps the new slice stay simple

  • DDD helps us choose the right slice.
  • Go helps keep the new slice operationally simple.
  • Small language surface and strong standard library.
  • Standard tooling, static binaries, and fast builds.
  • Straightforward deployment, clear concurrency, and Go 1 compatibility.
  • Use Go for maintainability and operational simplicity, not as a universal performance answer.

func (h BorrowBookHandler) Handle(ctx context.Context, cmd
                                BorrowBook) error {

                                copy := h.copies.Get(ctx, cmd.CopyID)
